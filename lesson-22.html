На прошлом уроке мы узнали, почему гигантские модели не помещаются в память GPU. Простого распределения вычислений часто не хватает: стандартный параллелизм данных (DDP) дублирует веса и состояния оптимизатора на каждой видеокарте. Этот урок посвящен продвинутым стратегиям экономии VRAM — технологии ZeRO, обучению со смешанной точностью и вытеснению активаций.

🔑 Концепция ZeRO (Zero Redundancy Optimizer): Разработанная Microsoft (в рамках библиотеки DeepSpeed), технология ZeRO полностью устраняет дублирование данных в памяти при параллелизме данных, не увеличивая при этом накладные расходы на связь. Она делит всю память на три этапа оптимизации.

1. Три этапа оптимизации ZeRO

  • ZeRO-Stage 1: Разделение состояний оптимизатора (Optimizer State Partitioning). Память под состояния Adam (которая занимает до 50-60% всей памяти) нарезается на равные куски между GPU. Каждая видеокарта обновляет только свою часть весов. Экономия VRAM: до 4х раз.
  • ZeRO-Stage 2: Разделение градиентов (Gradient Partitioning). Как только GPU вычисляет градиент для определенного слоя, этот градиент немедленно отправляется на ту карту, которая отвечает за данный кусок оптимизатора, и удаляется из локальной памяти. Экономия VRAM: до 8х раз.
  • ZeRO-Stage 3: Разделение весов (Parameter Partitioning). Сама модель полностью разрезается по слоям между GPU. Во время прямого и обратного прохода нужный слой динамически собирается через сетевую операцию All-Gather прямо перед вычислением, а затем мгновенно стирается из памяти чипа. Позволяет обучать модели на сотни миллиардов параметров.

2. Обучение со смешанной точностью (Mixed Precision Training)

Вычисления в формате FP32 (32 бита) избыточны для нейросетей. Обучение со смешанной точностью переводит большинство матричных умножений в быстрый 16-битный формат (FP16 или современный BF16, предотвращающий переполнение за счет широкого динамического диапазона, как у FP32).

⚠️ Проблема исчезновения под диапазон (Underflow): При использовании FP16 крошечные значения градиентов могут округлиться до абсолютного нуля, полностью сломав обучение. Для борьбы с этим применяется Градиентное масштабирование (Gradient Scaling): перед обратным проходом лосс умножается на большой коэффициент (например, 2¹⁶), выталкивая градиенты в безопасную зону типов данных, а перед шагом оптимизатора они делятся обратно. 
import torch
import torch.nn as nn

model = nn.Sequential(nn.Linear(4096, 4096), nn.ReLU(), nn.Linear(4096, 10)).cuda()
criterion = nn.MSELoss()
optimizer = torch.optim.AdamW(model.parameters(), lr=0.001)

# Инициализируем скалер градиентов для FP16
scaler = torch.cuda.amp.GradScaler()

inputs = torch.randn(32, 4096).cuda()
targets = torch.randn(32, 10).cuda()

optimizer.zero_grad()

# Включаем контекстный менеджер автоматического смешивания точности (AMP)
with torch.autocast(device_type='cuda', dtype=torch.float16):
    outputs = model(inputs)
    loss = criterion(outputs, targets)

# Вызываем обратный проход через скалер для защиты от Underflow
scaler.scale(loss).backward()

# Шаг оптимизатора выполняется только после демасштабирования градиентов
scaler.step(optimizer)
scaler.update()

3. Контрольные точки активаций (Activation Checkpointing)

Во время прямого прохода (Forward pass) нейросеть обязана сохранять в памяти VRAM выходы (активации) абсолютно каждого промежуточного слоя, чтобы использовать их при подсчете производных на обратном проходе. Для глубоких моделей с длинным контекстом эта память мгновенно переполняется.

Решение: Мы сохраняем в памяти активации только некоторых базовых слоев (например, каждого 4-го блока трансформера). Если обратному проходу требуются промежуточные активации, PyTorch динамически перевычисляет их «на лету», заново запуская короткий прямой проход для этого скрытого подблока. Это увеличивает время вычислений на 25-30%, но экономит до 60-70% памяти под активации.

from torch.utils.checkpoint import checkpoint

class DeepDenseBlock(nn.Module):
    def __init__(self):
        super().__init__()
        self.net = nn.Sequential(nn.Linear(2048, 2048), nn.ReLU(), nn.Linear(2048, 2048))

    def forward(self, x):
        # Оборачиваем вычисления в checkpoint. 
        # Теперь активации внутренних слоев этого блока не будут храниться в VRAM во время прямого прохода
        return checkpoint(self.net, x, use_reentrant=False)

import torch
import torch.nn as nn
from torch.utils.checkpoint import checkpoint

def create_deep_model(layers=20, dim=2048):
    layers_list = []
    for i in range(layers):
        # Каждый 4-й блок оборачиваем в checkpoint для экономии памяти
        block = nn.Sequential(nn.Linear(dim, dim), nn.ReLU(), nn.Linear(dim, dim))
        if i % 4 == 0:
            layers_list.append(lambda x, b=block: checkpoint(b, x, use_reentrant=False))
        else:
            layers_list.append(block)
    return nn.Sequential(*layers_list)

def benchmark_scenario(model, inputs, targets, scenario_name, use_amp=False, use_checkpoint=False):
    torch.cuda.reset_peak_memory_stats()
    
    if use_amp:
        scaler = torch.cuda.amp.GradScaler()
        with torch.autocast('cuda', dtype=torch.float16):
            outputs = model(inputs)
            loss = nn.MSELoss()(outputs, targets)
        scaler.scale(loss).backward()
    else:
        outputs = model(inputs)
        loss = nn.MSELoss()(outputs, targets)
        loss.backward()
    
    peak_mb = torch.cuda.max_memory_allocated() / 1024**2
    print(f"{scenario_name}: {peak_mb:.1f} MB")
    return peak_mb

if __name__ == "__main__":
    device = torch.device('cuda' if torch.cuda.is_available() else 'cpu')
    
    print("=== Benchmark памяти ===")
    # Сценарий 1: Базовый FP32
    model_fp32 = create_deep_model().to(device)
    inputs = torch.randn(8, 2048).to(device)
    targets = torch.randn(8, 2048).to(device)
    
    fp32_mem = benchmark_scenario(model_fp32, inputs, targets, "FP32 (baseline)", use_amp=False)
    
    # Сценарий 2: AMP FP16
    model_amp = create_deep_model().to(device)
    amp_mem = benchmark_scenario(model_amp, inputs, targets, "AMP FP16", use_amp=True)
    
    # Сценарий 3: AMP + Checkpoint
    model_cp = create_deep_model().to(device)
    cp_mem = benchmark_scenario(model_cp, inputs, targets, "AMP + Checkpoint", use_amp=True, use_checkpoint=True)
    
    print(f"\n=== Итоги ===")
    print(f"AMP экономия: {(1-amp_mem/fp32_mem)*100:.1f}%")
    print(f"Checkpoint доп. экономия: {(1-cp_mem/amp_mem)*100:.1f}%")
📚 Как работает Automatic Mixed Precision (AMP)

Два ключевых компонента AMP:

🔹 1. Автокаст (autocast)

torch.autocast — контекстный менеджер, который автоматически выбирает оптимальную точность для каждой операции:

  • Матричные умножения (GEMM) → FP16/BF16 (быстро, мало памяти)
  • Сведение к скаляру, деление, экспонента → FP32 (точно)
  • Активации сохраняются в той точности, в которой вычислены

🔹 2. Градиентный скалер (GradScaler)

Решает проблему underflow в FP16:

  1. Scale: Перед backward() лосс умножается на коэффициент (например, 2¹⁶ = 65536)
  2. Backward: Градиенты считаются в "увеличенном" масштабе, не обнуляясь
  3. Unscale & Step: Перед обновлением весов градиенты делятся обратно, и оптимизатор делает шаг
  4. Update: Скалер адаптирует коэффициент: если был NaN/Inf — уменьшает, если всё чисто — может увеличить
💡 BF16 vs FP16: BF16 (Brain Float 16) имеет такой же диапазон экспоненты, как FP32 (8 бит), но только 7 бит мантиссы. Это делает его устойчивым к переполнению при обучении, в отличие от FP16. Современные GPU (Ampere+, H100) поддерживают BF16 нативно.
