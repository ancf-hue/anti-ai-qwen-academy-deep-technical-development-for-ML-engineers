Добро пожаловать в заключительный Блок 5: Низкоуровневая оптимизация и Production. Современные LLM содержат десятки миллиардов параметров, и их физически невозможно обучить на одной видеокарте — модель просто не поместится в память GPU. Для решения этой проблемы инженеры используют распределенные вычисления на кластерах из сотен и тысяч GPU. На этом уроке мы разберем основные стратегии параллелизма и низкоуровневые примитивы связи.

🔑 Фундаментальное ограничение памяти: Каждый параметр в формате FP32 занимает 4 байта. Но в процессе обучения память расходуется не только на веса (Weights). Нам нужно хранить: градиенты (4 байта), состояния оптимизатора Adam (8 байт на параметр) и активации каждого слоя для обратного прохода. Итого, для модели на 7B параметров требуется минимум 120–140 ГБ VRAM, что значительно превышает объем флагманских GPU (H100/H200/B200).

1. Три стратегии параллелизма

  • Параллелизм данных (Data Parallelism / DDP): Модель полностью реплицируется на каждую GPU. Батч данных разбивается на части (микробатчи). Каждая карта считает свой лосс и градиенты независимо, а затем они синхронизируются через сетевую операцию All-Reduce. Применяется, если сама модель влезает в одну GPU.
  • Тензорный параллелизм (Tensor Parallelism / TP): Разрезает отдельные матрицы весов (например, внутри слоев Multi-Head Attention или FFN) по строкам или столбцам между несколькими GPU. Каждая видеокарта считает только кусок матричного умножения. Требует экстремально быстрой связи между чипами (NVLink).
  • Конвейерный параллелизм (Pipeline Parallelism / PP): Разрезает модель горизонтально по слоям. Например, первые 10 слоев трансформера выполняются на GPU 0, следующие 10 слоев — на GPU 1 и так далее. Сигнал передается последовательно по цепочке карт. Чтобы GPU не простаивали в ожидании друг друга (проблема "пузыря конвейера"), батч нарезается на микробатчи по алгоритмам 1F1B (One Forward, One Backward).

2. Низкоуровневые сетевые примитивы (Collective Communications)

Синхронизация данных на уровне кластера опирается на высокопроизводительные библиотеки связи (NCCL от NVIDIA). Основные операции:

  • All-Reduce: Собирает тензоры со всех GPU, суммирует (или усредняет) их и рассылает итоговый результат обратно на все GPU. Главный инструмент DDP.
  • All-Gather: Собирает локальные тензоры со всех карт и склеивает их в один большой тензор на каждой GPU.
  • Reduce-Scatter: Суммирует тензоры со всех GPU, но делит итоговый результат на части, отдавая каждой карте только её уникальный кусок.

3. Пример инициализации Distributed Data Parallel (DDP) в PyTorch

Для запуска кода на нескольких картах используется модуль torch.distributed. Процесс инициализации создает группу процессов, где каждая GPU идентифицируется уникальным номером (Rank).

import os
import torch
import torch.nn as nn
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

def setup_distributed_training(rank, world_size):
    # rank: уникальный ID текущего процесса (индекс GPU от 0 до world_size-1)
    # world_size: общее количество участвующих GPU в кластере
    
    os.environ['MASTER_ADDR'] = 'localhost' # IP-адрес главной машины
    os.environ['MASTER_PORT'] = '12355'     # Свободный порт для сетевой связи
    
    # Инициализируем бэкенд NCCL (рекомендован NVIDIA для GPU)
    dist.init_process_group("nccl", rank=rank, world_size=world_size)
    torch.cuda.set_device(rank)

def train_ddp_step(rank, world_size):
    setup_distributed_training(rank, world_size)
    
    # Создаем кастомную модель и отправляем её на конкретную GPU
    model = nn.Linear(10, 10).to(rank)
    
    # Оборачиваем модель в DDP
    # Теперь при вызове backward() градиенты будут автоматически синхронизироваться через All-Reduce
    model = DDP(model, device_ids=[rank])
    
    criterion = nn.MSELoss()
    optimizer = torch.optim.SGD(model.parameters(), lr=0.01)
    
    # Имитация прохода по уникальному микробатчу для этой GPU
    inputs = torch.randn(20, 10).to(rank)
    targets = torch.randn(20, 10).to(rank)
    
    outputs = model(inputs)
    loss = criterion(outputs, targets)
    
    optimizer.zero_grad()
    loss.backward() # Автоматический All-Reduce градиентов под капотом!
    optimizer.step()
    
    # Очищаем группу процессов после завершения обучения
    dist.destroy_process_group()

# В продакшене этот скрипт запускается через консоль:
# torchrun --nproc_per_node=2 distributed_script.py (для запуска на 2-х GPU)
⚠️ Ловушка DDP (DataLoader Sampler): Если при использовании DDP запустить стандартный DataLoader, все GPU получат абсолютно одинаковые батчи данных. Чтобы этого избежать, необходимо принудительно передавать в DataLoader специальный сэмплер: sampler=torch.utils.data.distributed.DistributedSampler(dataset). Он гарантирует, что каждая видеокарта будет читать только свою уникальную часть датасета.

    # Каждая GPU получает полный вектор [10, 14, 18, 22]

    # 2. Reduce-Scatter:
    # Считаем сумму по столбцам: [10, 14, 18, 22]
    # Затем SCATTER (разбиваем) этот результат:
    # GPU 0 получает элемент 0: 10
    # GPU 1 получает элемент 1: 14
    # GPU 2 получает элемент 2: 18
    # GPU 3 получает элемент 3: 22
    # Память на GPU: всего 1 элемент! (Экономия в 4 раза)
    
    print("All-Reduce: Каждый GPU хранит [10, 14, 18, 22]")
    print("Reduce-Scatter: GPU 0 хранит [10], GPU 1 хранит [14]...")

📚 DeepSpeed ZeRO: Zero Redundancy Optimizer

Как Reduce-Scatter спасает память?

В обычном DDP каждая GPU хранит полную копию весов, градиентов и оптимизатора. Это избыточно (Redundancy). Технология ZeRO устраняет эту избыточность, распределяя состояния оптимизатора между GPU с помощью Reduce-Scatter.

🔑 Стадии ZeRO:

  • ZeRO-1 (Optimizer States): Градиенты и состояния оптимизатора (Adam states) разбиваются (sharded) между GPU. При обновлении весов используется Reduce-Scatter. Память экономится в N раз (где N — число GPU).
  • ZeRO-2 (+ Gradients): Добавляется шардирование градиентов. Теперь и градиенты не дублируются на каждой карте.
  • ZeRO-3 (+ Parameters): Сами веса модели тоже шардируются. При прямом проходе GPU обмениваются кусками весов (All-Gather), перемножают и забывают. Это позволяет обучать модели размером в ТРИЛЛИОНЫ параметров.
💡 Почему это важно? Без ZeRO обучение модели на 70B параметров потребовало бы тысячи GPU только под память оптимизатора. С ZeRO-3 вы можете обучить такую модель на гораздо меньшем кластере, эффективно используя ресурсы.
