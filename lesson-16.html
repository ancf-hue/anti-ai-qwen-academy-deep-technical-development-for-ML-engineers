Добро пожаловать в Блок 4: Масштабирование и инженерия LLM. Нейросети не умеют напрямую читать буквы или слова. Чтобы передать текст в трансформер, его нужно разбить на фрагменты (токены) и сопоставить каждому токену уникальный числовой индекс из словаря. На этом уроке мы изучим алгоритмы построения словарей для современных LLM и напишем кастомный токенизатор.

🔑 Проблема наивных подходов: Токенизация по целым словам создает гигантские словари (миллионы слов), которые не вмещаются в память, и пасует перед опечатками. Посимвольная токенизация делает последовательность слишком длинной, перегружая квадратичную сложность механизма внимания. Современный стандарт — токенизация подслов (Subword Tokenization).

1. Алгоритм Byte Pair Encoding (BPE)

Алгоритм BPE — это фундамент токенизаторов в семействах моделей GPT, Llama и многих других. Он начинает с посимвольного (или побайтового) представления текста и итеративно объединяет самые частые пары элементов в один новый токен, пока словарь не достигнет заданного размера (Vocab Size).

Основные преимущества подсловных токенизаторов:

  • Эффективное сжатие: Частые слова (например, «привет») кодируются одним токеном, а редкие или новые слова разбиваются на знакомые корни и суффиксы.
  • Отсутствие OOV (Out-of-Vocabulary): Любое неизвестное слово или спецсимвол гарантированно могут быть деконструированы до базовых байтов, предотвращая фатальные сбои модели.

2. Реализация базового алгоритма BPE на чистом Python

Поймем изнутри логику тренировки BPE: подсчет пар и их слияние.

# Исходный тренировочный мини-корпус текстов
corpus = [
    "кот ест",
    "скат ест",
    "кот спит"
]

# Шаг 1: Инициализируем базовый словарь отдельными символами и пробелом
# Добавляем специальный маркер конца слова  для разделения контекста
vocab = set()
for text in corpus:
    for word in text.split():
        vocab.update(list(word) + [""])

# Представим корпус как списки символов
splits = {tuple(list(word) + [""]): count for text in corpus for word, count in [(w, text.split().count(w))]}

def get_stats(splits):
    # Подсчитываем частоту всех пар символов, идущих подряд
    pairs = {}
    for word, freq in splits.items():
        for i in range(len(word) - 1):
            pair = (word[i], word[i+1])
            pairs[pair] = pairs.get(pair, 0) + freq
    return pairs

def merge_vocab(pair, splits):
    # Сливаем самую частую пару в один токен во всем корпусе
    new_splits = {}
    bigram = re.escape(' '.join(pair))
    p = re.compile(r'(?<!\S)' + bigram + r'(?!\S)')
    for word, freq in splits.items():
        word_str = ' '.join(word)
        # Объединяем символы в строке и превращаем обратно в кортеж
        new_word = tuple(p.sub(''.join(pair), word_str).split())
        new_splits[new_word] = freq
    return new_splits

import re
# Запустим 5 итераций слияния (BPE Merges)
for i in range(5):
    pairs = get_stats(splits)
    if not pairs:
        break
    # Находим самую частую пару
    best_pair = max(pairs, key=pairs.get)
    splits = merge_vocab(best_pair, splits)
    print(f"Итерация {i+1}: Сливаем пару {best_pair} -> {''.join(best_pair)}")

3. Побайтовый BPE (Byte-level BPE)

Если текст написан на разных языках (мультиязычные модели), базовых символов Юникода становится слишком много, и они раздувают словарь еще до начала слияний. Модели типа GPT-2/3/4 и Llama используют Byte-level BPE: они переводят любую входящую строку в сырые байты (значения от 0 до 255). Начальный размер словаря всегда равен 256, и поверх этого запускается стандартный алгоритм BPE.

⚠️ Артефакты токенизации в LLM: Поскольку BPE жадно максимизирует частоту пар, добавление лишнего пробела перед словом полностью меняет его токенизацию. Например, токен для слова "apple" и токен для " apple" (с пробелом) — это два абсолютно разных индекса в словаре. Помните об этом при подготовке промптов.

import tiktoken

enc = tiktoken.get_encoding("cl100k_base")

text_en = "The quick brown fox jumps over the lazy dog"
text_ru = "Быстрая коричневая лиса прыгает через ленивую собаку"

# Кодируем тексты
tokens_en = enc.encode(text_en)
tokens_ru = enc.encode(text_ru)

# Декодируем обратно для наглядности
decoded_en = [enc.decode([t]) for t in tokens_en]
decoded_ru = [enc.decode([t]) for t in tokens_ru]

print(f"EN: {len(tokens_en)} токенов -> {decoded_en}")
print(f"RU: {len(tokens_ru)} токенов -> {decoded_ru}")
print(f"Русский текст требует в {len(tokens_ru)/len(tokens_en):.2f}x больше токенов")

# Вывод покажет, что русские слова дробятся на больше подслов
# из-за богатой морфологии и меньшего покрытия в тренировочных данных
📚 Почему русский язык «дороже» в токенах?

Морфология и покрытие словаря

Токенизаторы современных LLM (GPT, Llama) обучаются преимущественно на англоязычных данных. В результате:

🔑 Факторы «удорожания» русского текста:

  1. Богатая морфология: Русские слова имеют множество окончаний, приставок и суффиксов. Слово «бежать» порождает «бегу», «бежишь», «бежал», «бегущий» — каждый вариант может стать отдельным токеном или разбиться на части.
  2. Меньшее покрытие в BPE-словаре: Частые английские слова («the», «and», «ing») закодированы одним токеном. Их русские аналоги («и», «в», «-ть») часто разбиваются на 2-4 подтокена.
  3. Кириллица = больше байтов: В byte-level BPE каждый символ кириллицы занимает 2 байта в UTF-8, что увеличивает начальное количество элементов для слияния.
Пример:
EN: "running" → 1 токен [running]
RU: "бегущий" → 3 токена [б, ег, ущий]

Контекстное окно 4096 токенов = ~3000 английских слов ≈ ~1200 русских слов
💡 Инженерный вывод: При работе с русскоязычными промптами учитывайте, что лимит контекста исчерпывается в 2-3 раза быстрее. Для длинных документов используйте chunking (разбиение на части) или summarization (краткое изложение) перед подачей в модель.
