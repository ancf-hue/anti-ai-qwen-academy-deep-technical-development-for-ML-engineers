🔑 Концепция Динамического Бэтчинга (Dynamic Batching): В продакшене запросы от сотен пользователей приходят асинхронно и в случайные моменты времени. Если обрабатывать их по одному, ресурсы GPU будут расходоваться неэффективно. Специализированный сервер инференса перехватывает поток одиночных пользовательских запросов, упаковывает их в один общий батч за микросекундное окно ожидания (например, 5 мс), прогоняет через GPU за один такт и распределяет ответы обратно пользователям.

1. Индустриальный стек деплоя моделей

  • ONNX (Open Neural Network Exchange): Кроссплатформенный формат представления моделей в виде сериализованного графа вычислений, независимого от исходного фреймворка. Позволяет запустить модель, написанную на PyTorch, в среде C++, C# или Java.
  • NVIDIA TensorRT: Ультимативный компилятор инференса под GPU NVIDIA. Он перестраивает граф ONNX: объединяет слои, удаляет неиспользуемые узлы, автоматически подбирает оптимальные низкоуровневые ядра (CUDA/Triton) и может квантовать модель в INT8 с использованием калибровочного датасета. Выжимает максимум терафлопсов из железа.
  • Triton Inference Server: Промышленный оркестратор инференса от NVIDIA. Поддерживает одновременную работу моделей на нескольких GPU, умеет выполнять каскады моделей (Ensembles/Pipelines) и реализует тот самый Dynamic Batching. Применяется для классических архитектур (CNN, RNN, BERT, эмбеддеры).
  • vLLM: Высокопроизводительный движок, разработанный специально для инференса больших языковых моделей (LLM). Благодаря прорывной технологии PagedAttention (динамическое управление памятью KV-кэша по аналогии с виртуальной памятью в операционных системах) vLLM позволяет увеличить пропускную способность генерации текста до 24х раз по сравнению со стандартным пайплайном Hugging Face.

2. Экспорт кастомной архитектуры в формат ONNX

Процесс экспорта заключается в «проливании» фейкового тензора через модель для трассировки (Tracing) и фиксации всех выполненных математических операций в статическом графе.

import torch
import torch.nn as nn

class ProductionModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(512, 256),
            nn.ReLU(),
            nn.Linear(256, 10)
        )
    def forward(self, x):
        return self.net(x)

# Инициализируем модель и переводим в режим инференса
model = ProductionModel().eval()

# Генерируем "сигнальный" (dummy) входной тензор нужной формы
dummy_input = torch.randn(1, 512)

# Указываем пути сохранения и динамические оси
# По умолчанию размеры батча жестко прописываются в графе. 
# Чтобы модель принимала батчи любого размера в будущем, регистрируем dynamic_axes.
onnx_path = "compiled_model.onnx"

torch.onnx.export(
    model, 
    dummy_input, 
    onnx_path,
    export_params=True,        # Сохранять обученные веса внутри файла
    opset_version=17,          # Версия операторов ONNX
    do_constant_folding=True,  # Автоматически предвычислять константные операции
    input_names=['input_tensor'],
    output_names=['output_tensor'],
    dynamic_axes={
        'input_tensor': {0: 'batch_size'}, # Нулевая ось (батч) объявляется динамической
        'output_tensor': {0: 'batch_size'}
    }
)
print(f"Модель успешно скомпилирована и сохранена в отчуждаемый файл: {onnx_path}")

3. Проверка инференса на чистом C++ движке (ONNX Runtime)

Теперь мы можем полностью удалить PyTorch и запустить инференс модели в Python (или C++) через сверхлегкую библиотеку onnxruntime, которая работает напрямую с аппаратными ускорителями чипа.

import onnxruntime as ort
import numpy as np

# Инициализируем сессию инференса с привязкой к GPU (CUDA Execution Provider)
providers = ['CUDAExecutionProvider', 'CPUExecutionProvider']
session = ort.InferenceSession("compiled_model.onnx", providers=providers)

# Готовим входные данные в виде чистого массива NumPy
input_data = np.random.randn(32, 512).astype(np.float32) # Батч из 32 примеров

# Запуск вычислений графа
outputs = session.run(
    output_names=['output_tensor'], 
    input_feed={'input_tensor': input_data}
)

print("Форма выходных данных из ONNX Runtime:", outputs.shape) # [32, 10]
print("Вычисления выполнены без участия PyTorch и Python-оверхеда!")
⚠️ Главное правило Production-деплоя: Скомпилированные под TensorRT файлы движка (форматы .engine или .trt) жестко привязаны к микроархитектуре конкретной видеокарты. Файл, скомпилированный на видеокарте RTX 4090, физически откажется запускаться на сервере с картой A100 или H100. Всегда проводите TensorRT-компиляцию непосредственно на целевом сервере в процессе CI/CD сборки.

import torch
import torch.nn as nn
import onnxruntime as ort
import numpy as np

class CustomModel(nn.Module):
    def __init__(self, dim=512):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(dim, dim * 2),
            nn.GELU(),
            nn.Linear(dim * 2, dim)
        )
    
    def forward(self, x):
        return self.net(x)

def export_to_onnx(model, dummy_input, onnx_path):
    model.eval()
    torch.onnx.export(
        model, dummy_input, onnx_path,
        export_params=True,
        opset_version=17,
        do_constant_folding=True,
        input_names=['input'],
        output_names=['output'],
        dynamic_axes={'input': {0: 'batch'}, 'output': {0: 'batch'}}
    )
    print(f"✅ Экспортировано: {onnx_path}")

def validate_onnx_export(pytorch_model, onnx_path, test_input):
    pytorch_model.eval()
    with torch.no_grad():
        pt_output = pytorch_model(test_input).cpu().numpy()
    
    session = ort.InferenceSession(onnx_path, providers=['CPUExecutionProvider'])
    onnx_output = session.run(['output'], {'input': test_input.cpu().numpy()})[0]
    
    # Валидация с точностью до 4 знаков после запятой
    np.testing.assert_allclose(pt_output, onnx_output, rtol=1e-4, atol=1e-4)
    print("✅ Валидация пройдена: max diff =", np.max(np.abs(pt_output - onnx_output)))
    return True

def benchmark_inference(onnx_path, batch_sizes=[1, 8, 32], runs=100):
    """Бенчмарк производительности ONNX Runtime"""
    session = ort.InferenceSession(onnx_path, providers=['CUDAExecutionProvider'])
    
    for batch_size in batch_sizes:
        test_input = np.random.randn(batch_size, 512).astype(np.float32)
        
        # Warmup
        for _ in range(10): session.run(['output'], {'input': test_input})
        
        # Benchmark
        import time
        start = time.perf_counter()
        for _ in range(runs): session.run(['output'], {'input': test_input})
        duration = time.perf_counter() - start
        
        throughput = (runs * batch_size) / duration
        print(f"Batch {batch_size:3d}: {duration/runs*1000:6.2f} ms/req | {throughput:7.1f} req/sec")

if __name__ == "__main__":
    print("=== Финальный продакшн-пайплайн ===")
    
    # 1. Создаём и экспортируем модель
    model = CustomModel().eval()
    dummy = torch.randn(1, 512)
    export_to_onnx(model, dummy, "final_model.onnx")
    
    # 2. Валидируем корректность
    test_batch = torch.randn(8, 512)
    validate_onnx_export(model, "final_model.onnx", test_batch)

    benchmark_inference("final_model.onnx")

📚 PagedAttention в vLLM: Как работает прорыв для LLM

Проблема: Управление памятью KV-кэша в LLM

При генерации текста большие языковые модели хранят ключи и значения (KV-cache) всех предыдущих токенов для каждого запроса. В стандартных реализациях память под KV-кэш выделяется «с запасом» под максимальную длину контекста, что приводит к:

  • Фрагментации памяти (до 60-80% простаивает)
  • Ограничению количества параллельных запросов (batch size)
  • Невозможности эффективно обслуживать запросы разной длины

🔑 Решение vLLM: PagedAttention

Вдохновлённый системой виртуальной памяти в ОС, PagedAttention разбивает KV-кэш на фиксированные блоки («страницы» по 16-64 токена):

  1. Непрерывная логика, прерывистая физика: Модель «видит» непрерывную последовательность токенов, но физически KV-блоки могут быть разбросаны по памяти.
  2. Динамическое выделение: Новые страницы выделяются только когда они действительно нужны, а не заранее.
  3. Общий пул памяти: Все запросы в батче делят один пул страниц, что устраняет фрагментацию.
  4. Copy-on-Write: При параллельной генерации (beam search) страницы могут переиспользоваться без копирования.
🚀 Результат: vLLM достигает до 24x более высокой пропускной способности по сравнению с наивной реализацией Hugging Face, при этом сохраняя 100% точность генерации. Это делает его де-факто стандартом для продакшн-деплоя LLM.

📦 Когда что использовать:

  • Triton Inference Server: Классические модели (классификация, детекция, эмбеддеры), ансамбли, мульти-модальные пайплайны
  • vLLM: Большие языковые модели (LLM) с генерацией текста, чат-боты, RAG-системы
  • TensorRT-LLM: Максимальная производительность на NVIDIA GPU, когда важна каждая миллисекунда
