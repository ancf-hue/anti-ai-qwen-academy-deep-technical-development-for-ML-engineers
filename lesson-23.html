Когда вы создаете совершенно новую математическую архитектуру или кастомную функцию активации, высокоуровневых операций PyTorch может оказаться недостаточно. Стандартный код выполняет каждую операцию (например, сложение, а затем умножение) как отдельный вызов к памяти GPU. Это перегружает шину памяти. На этом уроке мы поднимемся на уровень Senior ML-разработки и научимся писать кастомные ядра, выполняющиеся напрямую на физических потоках видеокарты.

🔑 Архитектура памяти и потоков GPU: Видеокарта обрабатывает данные силами тысяч ядер параллельно. Потоки (Threads) группируются в блоки (Blocks), которые образуют сетку (Grid). Главный инженерный секрет оптимизации — минимизировать обращения к медленной глобальной памяти видеокарты (HBM) и максимально использовать сверхбыструю разделяемую память (Shared Memory) внутри блока потоков, объединяя операции вместе (Operator Fusion).

1. Индустриальный сдвиг: От сырого C++/CUDA к OpenAI Triton

Исторически кастомные ядра писались на C++ с расширениями CUDA, что требовало ручного управления регистрами, синхронизацией потоков (warps) и борьбы с утечками памяти. Современный золотой стандарт в ML-инженерии — язык OpenAI Triton. Он позволяет писать код на чистом Python, который компилируется в максимально производительные низкоуровневые ядра LLVM/CUDA, превосходящие ручной C++ код за счет автоматической оптимизации планирования блоков памяти.

2. Написание кастомного ядра поэлементного сложения векторов на Triton

Реализуем на Triton параллельное ядро, которое складывает два гигантских вектора. Каждый поток видеокарты будет обрабатывать один свой блок данных.

import torch
import triton
import triton.language as tl

@triton.jit
def add_kernel(x_ptr, y_ptr, output_ptr, n_elements, BLOCK_SIZE: tl.constexpr):
    # x_ptr, y_ptr, output_ptr — указатели на первый элемент массивов в памяти GPU
    
    # Находим глобальный ID блока потоков, который выполняет эту программу
    pid = tl.program_id(axis=0)
    
    # Вычисляем диапазон индексов элементов, за которые отвечает данный блок
    block_start = pid * BLOCK_SIZE
    offsets = block_start + tl.arange(0, BLOCK_SIZE)
    
    # Создаем бинарную маску, защищающую от выхода за границы массива
    mask = offsets < n_elements
    
    # Загружаем данные из медленной глобальной памяти GPU в быстрые регистры ядра
    x = tl.load(x_ptr + offsets, mask=mask)
    y = tl.load(y_ptr + offsets, mask=mask)
    
    # Производим математическую операцию параллельно на аппаратном уровне
    output = x + y
    
    # Записываем результат обратно в глобальную память GPU
    tl.store(output_ptr + offsets, output, mask=mask)

def triton_vector_add(x: torch.Tensor, y: torch.Tensor):
    # Обертка для вызова ядра из стандартного Python-кода
    output = torch.empty_like(x)
    assert x.is_cuda and y.is_cuda and output.is_cuda
    
    n_elements = x.numel()
    
    # Задаем размер блока (сколько элементов обрабатывает один программный блок)
    BLOCK_SIZE = 1024
    
    # Рассчитываем размер сетки грида (сколько всего блоков нужно запустить)
    grid = lambda meta: (triton.cdiv(n_elements, meta['BLOCK_SIZE']),)
    
    # Запуск ядра напрямую на GPU
    add_kernel[grid](x, y, output, n_elements, BLOCK_SIZE=BLOCK_SIZE)
    
    return output

# Проверка производительности ядра
x_gpu = torch.randn(1000000, device='cuda')
y_gpu = torch.randn(1000000, device='cuda')

# Тестируем кастомное ядро
res_triton = triton_vector_add(x_gpu, y_gpu)
res_pytorch = x_gpu + y_gpu

# Проверяем математическую точность совпадения результатов
assert torch.allclose(res_triton, res_pytorch)
print("Кастомное ядро Triton успешно выполнено и выдало абсолютно точный результат!")

3. Зачем нужен Operator Fusion (Слияние операторов)?

Представьте операцию внутри трансформера: out = Dropout(GeLU(LayerNorm(X))). В стандартном PyTorch данные трижды гоняются туда-обратно из регистров процессора в глобальную память VRAM. Написав кастомное ядро Fusion, мы загружаем блок данных X в локальную память потоков видеокарты один раз, последовательно применяем нормализацию, активацию, дропаут прямо внутри чипа и возвращаем в VRAM только готовый результат. Это убирает задержки шины и ускоряет инференс в несколько раз.

⚠️ Ограничение отладки ядер: Обычные инструменты отладки Python (вроде pdb или print) не работают внутри скомпилированных GPU-ядер. Ошибка в индексах внутри ядра может привести к падению драйвера видеокарты с ошибкой CUDA error: an illegal memory access was encountered. При написании ядер всегда проверяйте граничные маски.

import torch
import triton
import triton.language as tl
import time

@triton.jit
def relu_kernel(x_ptr, output_ptr, n_elements, BLOCK_SIZE: tl.constexpr):
    pid = tl.program_id(axis=0)
    block_start = pid * BLOCK_SIZE
    offsets = block_start + tl.arange(0, BLOCK_SIZE)
    mask = offsets < n_elements
    
    # Загружаем данные
    x = tl.load(x_ptr + offsets, mask=mask)
    
    # Применяем ReLU: max(0, x)
    output = tl.maximum(x, 0)
    
    # Сохраняем результат
    tl.store(output_ptr + offsets, output, mask=mask)

def triton_relu(x: torch.Tensor):
    output = torch.empty_like(x)
    assert x.is_cuda
    
    n_elements = x.numel()
    BLOCK_SIZE = 1024
    grid = lambda meta: (triton.cdiv(n_elements, meta['BLOCK_SIZE']),)
    
    relu_kernel[grid](x, output, n_elements, BLOCK_SIZE=BLOCK_SIZE)
    return output

if __name__ == "__main__":
    size = 50_000_000
    x = torch.randn(size, device='cuda')
    
    # Прогрев
    _ = triton_relu(x)
    _ = torch.relu(x)
    
    # Бенчмарк Triton
    torch.cuda.synchronize()
    start = time.perf_counter()
    res_triton = triton_relu(x)
    torch.cuda.synchronize()
    time_triton = time.perf_counter() - start
    
    # Бенчмарк PyTorch
    torch.cuda.synchronize()
    start = time.perf_counter()
    res_torch = torch.relu(x)
    torch.cuda.synchronize()
    time_torch = time.perf_counter() - start
    
    # Результаты
    assert torch.allclose(res_triton, res_torch)
    
    # Пропускная способность памяти (ГБ/с)
    bytes_processed = size * 4 * 2  # read + write, float32 = 4 bytes
    bw_triton = (bytes_processed / 1e9) / time_triton

📚 Memory Hierarchy GPU: От регистров до HBM

Иерархия памяти в GPU (от быстрой к медленной):

  • 🔹 Registers (~1 TB/s): Самая быстрая память, приватная для каждого потока. Хранит локальные переменные ядра. Ограниченное количество (~255 регистров на поток).
  • 🔹 Shared Memory (~80 TB/s): Разделяемая память внутри одного блока потоков (до 164 КБ на SM в H100). Ключ к оптимизации: загружаем данные из HBM один раз в Shared Memory, потом многократно используем.
  • 🔹 L1/L2 Cache: Автоматическое кэширование глобальных обращений. Прозрачно для программиста, но поведение можно тонко настраивать.
  • 🔹 HBM (High Bandwidth Memory, ~3 TB/s): Глобальная память видеокарты. Самая медленная из перечисленных, но самая ёмкая (до 80-141 ГБ в H100/H200). Все тензоры живут здесь.
💡 Почему Triton эффективнее ручного CUDA? Triton автоматически: 1) Оптимизирует размещение данных между регистрами и Shared Memory; 2) Генерирует оптимальные инструкции загрузки/сохранения; 3) Планирует выполнение блоков для максимизации occupancy. Вы пишете высокоуровневый код — компилятор делает низкоуровневую магию.

🔑 Правило оптимизации:

"Каждое обращение к глобальной памяти (HBM) должно приносить максимум полезных данных для вычислений." Именно поэтому мы используем большие BLOCK_SIZE и объединяем операции (Fusion) — чтобы загрузить данные один раз и сделать с ними всё необходимое.
